# 12306购票系统 - 项目分析与改进规划

## 一、项目概述

### 1.1 技术栈
- **后端**：Spring Boot 3.0.7 + Spring Cloud 2022.0.3 + Spring Cloud Alibaba 2022.0.0
- **前端**：React 19.2.1 + TypeScript + Vite 6.2.0 + Tailwind CSS
- **数据库**：MySQL + Redis
- **中间件**：Nacos（注册中心）、Canal（缓存一致性）
- **消息队列**：RocketMQ（未实现）
- **支付**：支付宝沙箱环境

### 1.2 微服务架构
```
Services/
├── gateway-service    # 网关服务 (端口: 8080) - WebFlux + Netty
├── ticket-service     # 票务服务 (端口: 8081)
├── seat-service       # 座位服务 (端口: 8082)
├── order-service      # 订单服务 (端口: 8083)
└── user-service       # 用户服务 (端口: 8084)
```

### 1.3 项目优先级
**功能 > 可靠性 > 性能 > 安全**

## 三、发现的问题与不足

### 3.1 可靠性问题（严重）

| 问题 | 描述 | 影响 |
|------|------|------|
| 分布式事务缺失 | 购票涉及多服务，缺乏事务保证 | 数据一致性问题 |
| 消息队列未实现 | 架构设计提到但未集成 | 异步处理、解耦能力不足，无法削峰填谷 |
| 无熔断降级 | 没有服务熔断和重试机制 | 雪崩风险高 |
| 幂等性未实现 | 重复请求可能产生问题 | 数据重复 |

### 3.2 性能问题

| 问题 | 描述 | 影响 |
|------|------|------|
| 选座架构问题 | Java内存计算+Lua脚本，高并发下大问题 | 选座性能瓶颈 |
| 缓存击穿风险 | 需要Canal监听binlog保证缓存一致性 | 缓存与数据库不一致 |
| 前端无代码分割 | 所有代码打包在一起 | 首屏加载慢 |

### 3.3 功能不足

| 问题 | 描述 | 影响 |
|------|------|------|
| 只支持直达搜索 | 无中转查询 | 用户选择少 |
| 无筛选功能 | 无法按时间、价格等维度筛选 | 用户体验差 |
| 无候补购票 | 高频需求缺失 | 高峰期抢票难 |
| 订单管理不完整 | 无取消/改签/退票、订单列表 | 功能不完善 |
| 无余票实时展示 | 余票状态不透明 | 用户决策困难 |

### 3.4 前端问题

| 问题 | 描述 | 影响 |
|------|------|------|
| 无路由管理 | 未使用React Router | URL和历史记录功能缺失 |
| TypeScript类型不完整 | 类型定义与后端API不对齐 | 类型安全差 |
| 表单验证简单 | 缺少完整的表单验证 | 用户体验差 |

| 问题 | 描述 | 影响 |

## 五、性能优化规划

### 5.1 选座架构优化

#### 当前问题
- Java内存计算选座策略 + Lua脚本原子性抢座
- 高并发下存在性能瓶颈

#### 优化方案（基于架构思路.txt）

**方案1：Redis BitMap优化**
- **余票数量缓存**：`(车次ID:日期:座位类型)` → 余票数组
  - 买A-C票扣减A-B、B-C的余票
  - 查询A-C票取A-B、B-C的最小值

- **余票详情缓存**：`(车次ID:日期:车厢号)` → BitMap座位占用
  - 每个座位用1bit表示占用状态
  - 区间A-C的占用 = A-B OR B-C

- **关键点**：
  - 使用Lua脚本保证原子性
  - BitMap压缩存储（1500座位 ≈ 188B）
  - 区间相邻拆分维护，简化计算

**方案2：数据库+Redis混合**
- 数据库只存已售座位
- Redis存热点数据
- 缺点：数据一致性问题

#### 推荐实现步骤

```lua
-- 选座Lua脚本示例
-- KEYS[1]: 余票数量key
-- KEYS[2]: 座位BitMap key
-- ARGV[1]: 开始区间索引
-- ARGV[2]: 结束区间索引
-- ARGV[3]: 座位索引

local remaining = redis.call('HGET', KEYS[1], ARGV[1])
if tonumber(remaining) <= 0 then
    return 0  -- 无余票
end

-- 检查并设置座位
local bitPosition = ARGV[3]
local currentBit = redis.call('GETBIT', KEYS[2], bitPosition)
if currentBit == 1 then
    return 0  -- 座位已被占用
end

redis.call('SETBIT', KEYS[2], bitPosition, 1)
redis.call('HINCRBY', KEYS[1], ARGV[1], -1)
return 1  -- 选座成功
```

#### 选座策略（Java端）
```
SeatStrategy (接口)
├── GreenCarStrategy          // 绿皮车策略
├── HighSpeedFirstStrategy    // 高铁一等座
├── HighSpeedSecondStrategy   // 高铁二等座
├── BusinessClassStrategy     // 商务座策略
└── SleeperStrategy           // 卧铺策略

SeatPreferenceStrategy
├── ContinuousStrategy        // 连续座位
├── SameCarriageStrategy      // 同车厢
└── WindowPreferenceStrategy  // 靠窗偏好
```

### 5.2 缓存一致性优化

## 六、可靠性提升规划

### 6.1 分布式事务

**方案**：Seata AT模式
- 购票流程涉及：ticket-service → seat-service → order-service
- 使用Seata保证跨服务事务一致性

### 6.2 消息队列集成

**选型**：RocketMQ
- 支持事务消息
- 保证消息不丢失（同步刷盘）
- 有重试和死信队列

**应用场景**：
1. **削峰填谷**：高峰期购票请求进入MQ，异步处理
2. **异步选座**：购票请求→MQ→座位服务→回调→订单服务
3. **弹性架构**：
   - 非高峰期：同步直接返回
   - 高峰期：异步处理+状态轮询

### 6.3 熔断降级

**方案**：Sentinel
- 接口级限流规则
- 熔断策略（慢调用比例、异常比例）
- 降级预案（返回默认值/排队等待）

### 6.4 幂等性设计

**方案**：
- 每个请求生成唯一ID（RequestId）
- Redis存储处理状态：`idempotent:{requestId}`
- 分布式锁防止重复处理

## 十二、附录：架构设计参考

### 12.1 选座数据结构设计

#### 维度分析
```
车次: ~10000
日期: 30天
区间: 3368个站点 → 约900万个区间组合
车厢: 15
座位: 118 → 约1500个座位/车
```

### 12.3 弹性架构设计

#### 非高峰期
```
购票服务 → 座位服务 → 订单服务
           ↓
      同步直接返回
```

#### 高峰期
```
购票服务 → MQ → 座位服务 → MQ → 订单服务
                ↓              ↓
             回调消息      标记成功
                ↓
         购票服务轮询状态 → 返回结果
```

#### 前端适配
```
收到响应 → 有结果 → 渲染
         → 无结果 → 加载动画 → 轮询状态查询
```
